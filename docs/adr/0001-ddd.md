# ADR 0001 — Uso de Domain-Driven Design (DDD)

## Estado
Aceptado

## Fecha
2026-03-28

## Contexto
El sistema de gestión de parqueaderos tiene un dominio rico con reglas de negocio complejas:
tarifas por tipo de vehículo, sesiones de parqueo, pagos y reportes de ocupación.
Se necesita un modelo que refleje fielmente el lenguaje del negocio y facilite
la evolución independiente de cada concepto.

## Decisión
Se adopta **Domain-Driven Design (DDD)** como enfoque de modelado. El paquete
`domain/` contiene únicamente modelos, puertos y servicios de dominio puros,
sin dependencias de frameworks de infraestructura.

### Elementos aplicados
- **Aggregates**: `ParkingSession` es el agregado raíz del proceso de parqueo.
- **Value Objects**: `Tariff`, `Money` representan conceptos inmutables.
- **Domain Events**: `VehicleEnteredEvent`, `VehicleExitedEvent`, `PaymentProcessedEvent`.
- **Ports (interfaces)**: definen los contratos que la infraestructura debe cumplir.

## Consecuencias
- El dominio es testeable de forma unitaria sin levantar contexto de Spring.
- Los cambios de infraestructura (BD, mensajería) no afectan las reglas de negocio.
- Requiere disciplina del equipo para no introducir anotaciones de frameworks en `domain/`.
